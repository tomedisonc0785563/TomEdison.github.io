On-Premises to AWS Migration Project

Migration of a Workload running in a Corporate Data Center to AWS using the Amazon EC2 and RDS service
VPC | AWS EC2 | AWS RDS | Internet Gateway | Git Bash | MySQL | Rehosting (Lift-and-Shift)


  • View in GitHub

Problem Statement

ABC company wants to migrate their production environment to the cloud with the same configurations and operating systems.
My Mission: The mission of my project is to deploy the application to the AWS cloud with minimal changes using rehosting(lift and shift) method, ensuring that all functionalities operate seamlessly in the new environment. I acted as the Cloud Specialist responsible for migrating a workload running in a Corporate DataCenter to AWS.

Introduction

Rehosting (lift and shift) is the process of migrating your on-premises application to the cloud without modifying it. This strategy is used mostly to migrate large-scale applications to satisfy specific business goals, such as launching a product in an accelerated timeline or leaving an on-premises data centre. The applications are rehosted on Amazon Elastic Compute Cloud (Amazon EC2) Windows instances that meet the requirements of the applications you migrate.

Planning and Strategy

The process begins with Planning, where sizing and prerequisites are determined to ensure a smooth transition. With the help of Excel, we can compare and conclude the best matching the on-premises and cloud-provided servers, versions, sizes and names. Next is the Implementation phase, focusing on deployment and adhering to best practices for optimal performance. During the Go Live phase, the migration undergoes validation through a dry run, followed by the final migration, also known as cutover. Post-migration, the post-go-live phase ensures the stability of the migrated workloads and provides ongoing support to address any issues. This structured approach helps in achieving a successful and efficient migration to the cloud.

Solution Architecture

This solution architecture illustrates the migration of an application from an on-premises corporate data centre to the AWS cloud. The architecture is designed to ensure a seamless transition while maintaining the functionality and performance of the application. The workload that is both the application server and database server are deployed in only a single availability zone since in the rehost method we don't take full advantage of the performance, scalability, and resiliency options of the AWS Cloud but migrate with minimum effort with no architectural changes, reduced cost and better compliance & security.

Step by step

Create and setup environment and resources

Created a public subnet as planned and 2 private subnets(since the RDS prerequisite needs 2 subnets). Also, created an internet gateway and attached our VPC along with a route table for the internet gateway to route to the public internet.


Launch the EC2 instance corresponding to the app server with a matching instance size and the right version of the operating system. Also, set a firewall with the security group inbound rules.


Create RDS MySQL corresponding to the data server version as planned in the same region that the app server was created.


Remotely connect to DB using GitBash, import and insert all the data. Then import the application to EC2 using Gitbash and run the application using GitBash.


Test results and outcomes

Try running the application in the browser.


Do the testing of the app. Check all the operations and test if the new integration has affected the application in any way. If we can see the articles section on the webpage then it means the database connection is working properly. Try inserting operations on articles.


Insert operations seem successful. The migration hasn't affected the system and the system is working seamlessly.


Conclusion

The rehosting method only has minimal effect on the application. This helps to avoid any unexpected issues that could have occurred due to the change in version size or type or other configuration of the resources during migration. We can later consider updating the configurations as needed after the successful migration.
