Autoscalable web application deployment

Scalable web application based on cpu utilization using Amazon Elastic Beanstalk, DynamoDB and Edge Locations

AWS VPC | AWS Elastic Beanstalk | Amazon EC2 | CloudFront | Elastic Load Balancer | Auto Scaling Group | DynamoDB


  • View in GitHub

Problem Statement


My Mission: The mission of my project is to implement scalability in the cloud for a web application that needs high availability as well.
I acted as the Cloud Solution Architect responsible for implementing a web application that needed support for the high demand of a large number of users accessing it simultaneously. This application has been used in large conferences that had more than 10,000 people, in-person and online, with participants from all over the world.

Introduction

This project focuses on implementing a robust, scalable web application using advanced AWS technologies. Leveraging AWS Elastic Beanstalk for seamless deployment and management, the application employs auto-scaling to handle varying loads efficiently at the lowest cost. Amazon CloudFront is utilized to deliver content globally with low latency through its edge locations. The application ensures high performance and availability by integrating Amazon DynamoDB for fast and flexible NoSQL database operations. This setup provides a comprehensive and scalable solution optimized for performance and reliability.

Solution Architecture

More than 10,000 people are expected to participate, both in-person and online. This event will be broadcast over the internet and inperson. At some point during the event, 10 vouchers will be drawn for 3 Cloud certifications. The audience will register the email to guarantee participation in the raffle. The objective of this project is to create an application with the necessary components for the participants to successfully access the page. For this, we will rely on AWS. We are going to use Elastic Beanstalk to deploy a web application and DynamoDB to store the emails. Let's rely on CloudFront to cache static and dynamic files to an Edge Location close to the user. This application will need to be responsive for Desktop and Mobile access.

We will deploy the application in an S3 bucket and share this source path with Elastic Beanstalk. DynamoDB will be used to store the participant's email ID. Elastic Beanstalk will be set up to create 2 instances and whenever CPU utilization reaches 50% then create a new instance until it reaches the maximum limit provided. Similarly, decrement the number of instances when CPU utilization goes down having a limit to a minimum number of instances. Amazon CloudFront will be used to cache the web application in all the edge locations available for high availability and cloudwatch will be used to monitor and log the events.

Step by step

Create and setup environment and resources

Created DynamoDB database table to store emails that gets from the application registration page input field.


Created Amazon Elastic Beanstalk(EBS) environment with following setups:

  • use default VPC
  • use the public url of S3 with application code as source code location in EBS
  • set up to have minimum instances 2 and maximum 4, with general purpose(SSD) instance type and 10GB storage
  • scaling must trigger respective to CPU utilization, when it reaches 50% increment 1 instance and decrement 1 when it goes down to 40%

As soon as amazon EBS is launched we can see instances running.


We can also access the application from domain url of EBS.


Test results and outcomes

We are not able to register email. Request the log file for each instances from EBS dashboard and find the errors being logged. Since instance is not given access to DynaamoDB this error was showing.


Attach the DynamoDB access to the IAM role of instance profile attached to EBS.


Registration process is able to complete successfully now. This should mean that input is passed to DynamoDB successfully.


To check, we can see that input text is inserted to DynamoDB table.


Create CloudFront to cache the web application in edge locations and allow all HTTP methods.


Monitor the health of instances using EBS management console dashboard.


When stress given to instance CPU utilization reaches 100% and health warning will be shown. Also, we can see the changes logged in events.


Auto scaler added one new instance under same load balancer. Since the instance CPU utilization reached above 50%, autoscaler was triggered to add one new instance as provided.


Currently we can see all the three instances running.


When the cpu utilization went down to 40% the instances scale down by one.


Conclusion

Amazon Elastic Beanstalk management console dashboard helps to monitor the events and to monitor the health of the instances. AWS CloudFront helps to cache the web application for quick access with low-latency and also handles the security of the website.
